BayWatch PLAN

A self-hosted drive-bay health & locator dashboard for the HPE Gen9 fleet (DL380 + DL360, Smart HBA / P440ar in HBA mode). Live LED-accurate chassis view, ZFS+SMART health, click-to-locate. Drafted from a three-model design pass (Codex - architecture, Gemini - UX/visualization, Cursor - risk/MVP), reconciled against the real hardware.

Where this came from. We already proved (live, on both servers) that the caddy amber fault and blue locate LEDs are controllable in HBA mode via the kernel SES interface, and a systemd health service is deployed that lights amber on unhealthy drives. This plan turns that proven engine into a real app with a web UI.

1 - The one constraint that shapes everything

The LED hardware lives on the bare-metal Proxmox hosts. The Kubernetes cluster runs as Talos VMs on those hosts, so a pod cannot see /sys/class/enclosure or /dev/sg*. Therefore the system is necessarily two-tier:

   browser ──https──>  [ BayWatch UI ]                         (Kubernetes pod, *.iacob.uk)
                         aggregator + SVG chassis + SSE
                              │  LAN (Cilium → 192.168.1.x)
              ┌───────────────┴────────────────┐
        https/token                      https/token
              ▼                                ▼
   [ bw-agent @ DL380 ]              [ bw-agent @ DL360 ]       (systemd, root, bare metal)
   owns SES + health + LEDs          owns SES + health + LEDs
   /sys/class/enclosure, sg_ses      zpool, smartctl

The agent is the only thing that touches hardware. The K8s app is a pretty, aggregating window onto the agents - if it dies, the health LEDs keep working.

2 - Key design decisions the load-bearing ones

D1 - Anchor bays on the SES element, not the block device (resolves Cursor's #1 risk)

Cursor's sharpest point: mapping a bay via its current /dev/sdX breaks exactly when a drive dies or is pulled - the moment you most need to find it. The hardware gives us a better anchor, and we verified it live:

  • Every bay is a stable SES element with a fixed slot number, under an enclosure with a unique logical_id (e.g. 0x50014380435ccf80).
  • An empty bay still exists as a controllable element - we saw DL380 enc 2:0:8:0 slot 6 report status: not installed yet remain lightable. So we can locate a dead/pulled drive's bay.
  • The slot attribute also fixes the silkscreen mismatch we hit (elements 4-7 → physical slots 8,7,6,5).
Anchor = (enclosure logical_id, slot) for the bay/LED. Correlate the drive in it by serial/WWN for health + labels. A persisted element ↔ last-seen serial map means a vanished ZFS member still resolves to a physical bay.

D2 - One writer per host owns all SES writes (resolves Cursor's concurrency race)

The current systemd health script and a locate feature would otherwise race on the same SES bits (health write clobbers the locate bit every 60s). Fix: fold health polling + locate into a single bw-agent daemon that holds desired LED state in memory and reconciles it atomically. The old timer/script is retired into the agent.

D3 - Security: LAN-bound, token-auth, writes limited to time-boxed locate (Cursor's #1 concern, pragmatic middle)

Cursor is right that a root HTTP daemon on the hypervisor is real attack surface. For a trusted home LAN the pragmatic stance:

  • Agent binds to the host LAN IP only; bearer-token auth (shared secret in a K8s Secret).
  • Reads are cheap; the only mutating endpoint is locate, and it is time-boxed (auto-clears, e.g. 60-300s) so nothing can pin a blue LED or wedge state.
  • Fault/health LEDs are computed solely by the agent from ZFS+SMART - never settable from the network.
  • Documented harder option for later: drop the HTTP write path entirely and have the pod invoke locate over an SSH forced-command key. (Deferred - token+LAN is enough for v1.)

D4 - The health engine stays independent of the UI

The agent's reconcile loop is self-contained: hardware truth = ZFS + SMART, evaluated on the host. The K8s app going down, or a network partition, never affects whether a failing drive shows amber. The UI degrades gracefully with a data-freshness badge.

3 - Visualization: generated SVG chassis (Gemini's rec, option A)

Three options weighed: (A) an SVG schematic generated from the real SES topology, (B) HPE product photos, (C) hybrid. Pick A. It is accurate (driven by live slot data), scalable, themeable to the glassmorphism Homepage vibe, and carries no copyright baggage. Photos can't map bays reliably and look wrong the moment a layout differs.

Each bay is a cell that renders one of four states, hover = drive detail (model/serial/size/temp/SMART/ZFS vdev), click = locate toggle:

healthy   fault (ZFS/SMART)   locate (pulsing)   empty bay. Locate pulses; everything respects prefers-reduced-motion.

5 - Shape of the build (Codex, trimmed to Cursor's MVP)

Stack

  • bw-agent: single static Go binary + systemd unit. No runtime deps, trivial to drop on Proxmox. Reconcile loop (poll ZFS+SMART, diff, apply SES), REST + SSE, token auth.
  • bw-ui: single Go binary serving an embedded SVG/JS frontend (one container image, app-template friendly). Aggregates agents over the LAN, streams changes to the browser via SSE. Deployed via bjw-s app-template HelmRelease + Gateway API HTTPRoute on baywatch.iacob.uk (envoy-internal), optionally behind Authentik.

"Events from the controller"

SES has no push. The agent fast-polls (~2-5s), diffs, and emits only changes upstream over SSE; a udev watch triggers an immediate refresh on drive insert/remove. To the browser it feels event-driven.

API sketch 
bw-agent
  GET  /v1/enclosures        → cages, slots, drive identity, health, led state
  GET  /v1/stream            → SSE: {slot, state} change events
  POST /v1/locate            → {enclosure_id, slot, seconds} (time-boxed, auto-clear)
  GET  /v1/healthz
bw-ui (K8s)
  GET  /api/fleet            → merged snapshot of both agents
  GET  /api/stream           → fan-in SSE to browser
  POST /api/locate           → proxied to the owning agent

6 - Phasing

MVP ~1 week, real value, low blast radius

  • bw-agent on both hosts: reconcile health→amber (port the proven logic), expose read API + SSE, one time-boxed locate endpoint, token auth, LAN-bound.
  • Persist the (enclosure_id, slot) ↔ serial map; handle empty/missing bays.
  • bw-ui: read-only live SVG chassis for both servers, hover details, click-to-locate with countdown.
  • Deploy bw-ui via app-template at baywatch.iacob.uk; add to Homepage.

v1

  • udev hot-plug, locate leases + conflict handling, data-freshness badges, audit log of LED writes.
  • Drive detail drawer (temp trend, SMART attributes, ZFS vdev/pool), per-cage health summary.

v2 defer unless wanted

  • Authentik SSO, mTLS or SSH-forced-command write path, Prometheus metrics, alert routing, "blink-on-failure" auto-locate.
